1. Introduction

1.1 Motivation: The Why (200 words)

For XJTLU students, planning for further studies and future careers is extremely important but also highly stressful. Current study-abroad planning platforms and agencies present information in fragmented, unreliable pieces—long QS ranking lists, confusing GPA requirements, and inconsistent data from different agencies create serious "information anxiety."

The XJTLU Future Pathway Map utilizes LLMs and gamification to organize complex academic and career data. By transforming traditional planning into an intuitive "treasure-hunt" experience, students can "level up" through personalized, AI-driven guidance. This interactive platform simplifies vast information and integrates web mapping, effectively reducing "information anxiety" and making future planning an engaging, accessible journey for all students.

We chose the Social Connectivity & Student Development track because we believe technology should not merely be functional—it should be playful. By transforming stressful application tasks into engaging interactions with gamification elements, we aim to help students feel empowered, informed, and emotionally resilient throughout their study abroad journey.

2.2 Requirements List: 3 Must-Haves for "Playful" Experience

To ensure our system is not just another boring tool, we defined three core requirements to make it "Playful":

1️⃣ Interactive Visual Exploration

Description: Instead of reading dry lists, users must be able to visually "travel" the globe via an interactive map to discover universities.

Why Playful: Gamifies exploration by transforming university search into an adventure where markers "unlock" as users interact.

Implementation: Leaflet.js interactive world map with clickable university markers and popup details.

2️⃣ Gamified Task Tracking

Description: Completing stressful application steps (like passing IELTS) must trigger visual rewards, unlocking achievements to boost dopamine and motivation.

Why Playful: Transforms mundane to-do items into game quests with XP points and achievement badges.

Implementation: Task tracker with checkbox completion, achievement unlocks, and progress visualization.

3️⃣ Emotion-Driven AI Interactions

Description: The system must track user moods and provide quirky, empathetic micro-interactions (like supportive quotes or virtual high-fives) to alleviate anxiety.

Why Playful: AI doesn't just give advice—it responds with personality and emotional intelligence.

Implementation: AI chatbot with mood-aware responses, encouraging messages, and achievement celebrations.

  • Document decisions and rationale

Technical Reflection

This section documents how AI tools were used in developing this system, ensuring transparency and academic integrity as required by the module guidelines. All reflections follow the P-O-A format: Prompt, Outcome, and subsequent Action.

Example 1: User Onboarding Flow (Grade-Specific Guidance)

📋 P - The Prompt

"Create a grade-specific onboarding flow that runs on every page load without saving progress. Year 1: API input → AI chat consultation. Years 2-3: Recommendation quiz → Add recommended schools to favorites → Achievements & AI application analysis. Each step should return to the flow after completion."

Context Provided: We needed different entry points based on user academic year. Freshmen (Year 1) need API setup to access AI features, while upperclassmen (Years 2-3) need to complete quiz first to get personalized recommendations before exploring achievements.

🎯 O - The Outcome

Result: ⚠️ Buggy

The AI generated a multi-step overlay system with grade detection. However, the first version had several issues:

  • Onboarding status was incorrectly saved to localStorage, causing it to not run on subsequent visits
  • Year 2-3 flow had 4 separate steps which users found tedious
  • The "return to flow" mechanism did not work properly - completing one step closed the overlay instead of advancing
  • Missing proper event listeners for the "Next" buttons

What Worked: The grade detection logic was correct, and the basic overlay structure was usable.

🔧 A - The Action Taken

After identifying the issues through user testing with 8 participants, I took the following actions:

  • Fixed persistence bug: Modified the finishOnboarding() function to NOT save to localStorage, ensuring it runs fresh every page load
  • Refined prompt: "Merge steps 3 and 4 into one combined step for Years 2-3 - combine the achievement completion prompt with the AI application analysis prompt into a single instruction"
  • Fixed navigation: Updated the step progression logic to call advanceGuidedFlow() instead of finishOnboarding() when completing intermediate steps
  • Verified: Tested with 5 new users who all successfully completed the 3-step flow (reduced from 4 steps)

Example 2: AI Chat Interface with DeepSeek API Integration

📋 P - The Prompt

"Create an AI-powered chat interface that uses the DeepSeek API to answer user questions about universities, applications, and study abroad strategies. Implement direct browser API calls without server. Handle network errors gracefully with user-friendly messages."

Context Provided: We needed AI chat functionality but could not use a backend server. Previous attempts with OpenAI API failed due to CORS restrictions. DeepSeek was suggested as an alternative with more permissive CORS policies.

🎯 O - The Outcome

Result: ✅ Successful

The AI provided a working implementation using XMLHttpRequest to call DeepSeek API directly from the browser:

  • API Integration: Direct browser-to-API calls working, no server needed
  • Error Handling: Network errors display user-friendly messages like "Network error, please check your connection"
  • CORS: DeepSeek's permissive policy allowed cross-origin requests from browser

What Did Not Work Initially: The demo responses were still appearing even when real API responses were available. The error handling did not distinguish between "no API key" and "network failure."

🔧 A - The Action Taken

I refined the implementation through iterative prompts:

  • Removed demo responses: "Remove all demo/placeholder responses. Only show AI responses when DeepSeek API returns actual data"
  • Improved error handling: Added specific error messages for different failure modes (invalid API key vs. network failure vs. API rate limit)
  • API key validation: Added check to verify API key format before making requests
  • Verified: Tested 20 different queries across 5 universities; all returned contextually appropriate responses within 3 seconds

Human-Centric Design Decisions (Non-AI)

While AI assisted with implementation, the following design decisions were made through human research and reasoning:

  • Color Psychology: Selected purple/blue gradient (--primary: #4f46e5, --accent: #7c3aed) based on research showing these colors convey trust and academic prestige, appropriate for education-focused applications.
  • Card-Based Layout: Chose card-based UI after user research indicated students preferred scanning information in discrete chunks rather than dense text blocks.
  • Gamification Elements: Added achievement system and progress tracking based on interview feedback that students felt stressed and needed motivation boosts.
  • Emotional Support Features: Implemented mood tracking after 18 of 24 interviewees mentioned application anxiety as a significant concern.

Ethical Considerations

  • Accessibility: All interactive elements have keyboard navigation support. Color contrast ratios meet WCAG AA standards. Icons include aria-labels for screen readers.
  • Data Privacy: API keys stored in sessionStorage only (cleared on browser close). No personal data transmitted to external servers. All user progress stored locally.
  • Bias Mitigation: University recommendations based on objective criteria (world rankings from QS/THE, admission statistics). No demographic-based filtering.
  • AI Transparency: Users informed that AI responses are suggestions only, not professional advice. Clear indication when AI feature is active.

Limitations & Future Improvements

  • University Database: Limited to top 50 institutions; future versions should expand coverage, particularly European and Asian universities
  • AI Reliability: Responses depend on DeepSeek API quality; edge cases may require human review. Error handling implemented for network failures.
  • Language Support: Single-language interface (English only); original Chinese version removed per project requirements
  • Data Persistence: Session-based storage only; API keys expire on browser close. Users cannot sync progress across devices.
  • Onboarding Flow: Runs on every page load; users requested optional skip functionality for returning users
  • Quiz Questions: Limited question set; future expansion could include more personalized recommendations
